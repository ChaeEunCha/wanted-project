# PRD — 원티드 API 기반 신입 구직자 대시보드

> 이 문서는 [PLANNING.md](./PLANNING.md)에서 정리한 기획 내용을 제품 요구사항 형태로 재구성한 문서입니다. 기능별 배경/근거는 PLANNING.md를 참고하고, 여기서는 요구사항·우선순위·완료 기준을 다룹니다.

## 1. 개요

- **서비스명**: 원티드 API 기반 대시보드
- **한 줄 정의**: 원티드 오픈API 데이터를 재구성해서, 처음 구직하는 청년이 겪는 정보 비대칭과 파편화 문제를 줄여주는 개인화 대시보드
- **데이터 소스**: Wanted OpenAPI v1/v2 (client_id/secret 인증). 일부 기능은 자체 DB(회원/프로필/지원상태)로 보완

## 2. 배경 및 문제 정의

처음 구직자는 아래 문제를 겪는다:

1. 신입 태그 공고인데 실제 요구 경력이 1~3년인 경우가 많아 헷갈림
2. 공고 속 기술 스택/직무 용어 장벽
3. 공고 홍수 속에서 마감일을 놓침
4. 지원 이력이 여기저기 파편화됨
5. 회사 정보(연봉/복지)를 알기 어려움
6. 내 역량이 공고 기준에 맞는지 판단이 어려움
7. (경쟁강도, 합격률을 모르는 문제도 있었으나 API 검토 결과 충족 불가 — 6절 참고)

## 3. 목표

- 신입 구직자가 **가짜 신입 공고를 걸러내고**, **용어를 몰라도 지원 여부를 판단**할 수 있게 한다
- 지원 과정(관심→지원→결과)을 **한 곳에서 관리**할 수 있게 한다
- 지원 전에 **회사 정보(연봉/복지)**를 확인할 수 있게 한다
- 내 역량과 공고 요구사항의 **갭을 액션으로 연결**한다

## 4. 타깃 유저 / 페르소나

- **주 타깃**: 경력 0년(완전 신입) 구직자
- **부 타깃**: 1~2년차 주니어 (경력 필터/매칭 로직 범위에 포함하되 UI 우선순위는 신입에 맞춤)

## 5. 스코프 — 제작 기능 (8개, 우선순위 순)

우선순위 기준: **0번·1번은 병렬 최우선 착수** (하나는 계정/DB 인프라, 하나는 API 연동 코어이며 이후 모든 기능이 이 둘에 의존).

| 우선순위 | 기능 | User Story | 완료 기준 (Acceptance Criteria) |
|---|---|---|---|
| P0 | 회원가입 & 프로필/포트폴리오 | 신입 구직자로서, 내 경력(0~2년)과 관심 태그·포트폴리오를 등록해서 이후 기능들이 나를 알아보게 하고 싶다 | 회원가입/로그인 가능, 경력 0~2년 범위 선택 UI, 관심 직군/스킬 태그 등록, 포트폴리오 링크/파일 등록. 이 데이터가 DB에 저장되어 4·6·3번 기능에서 재사용됨 |
| P0 | 신입 맞춤 필터 대시보드 | 신입 구직자로서, 진짜 신입이 지원 가능한 공고만 보고 싶다 | `annual_from`/`annual_to` 파싱해 "진짜 신입 가능" 여부 뱃지 표시, 직군/지역 조합 필터 동작. 자격요건 뱃지 추출 상세 스펙은 5-1절 참고 |
| P1 | 회사 정보(연봉/복지) | 구직자로서, 지원 전에 이 회사의 평균연봉과 복지를 확인하고 싶다 | 공고 상세에서 `company.registration_number` → `/insight/company` 체이닝으로 `averageSalary`/`hiredSalary` 표시, `detail.benefits` 텍스트 + `attraction_tags` 표시. `/insight/company` 503 응답 시 "정보 없음" 폴백 처리 |
| P2 | 용어 툴팁 | 구직자로서, 공고 속 낯선 기술/직무 용어에 마우스를 올리면 쉬운 설명을 보고 싶다 | `skill_tags` 각각에 툴팁 UI, 자체 구축한 용어 사전 데이터와 매칭되어 설명 노출 (매칭 안 되는 태그는 툴팁 미노출) |
| P3 | 지원 여정 칸반보드 | 구직자로서, 내가 관심 있거나 지원한 공고를 단계별로 관리하고 싶다 | 관심/준비중/지원함/결과대기 4단계 칸반, 카드에 회사명·마감일·스킬태그 스냅샷 표시, 드래그 또는 버튼으로 단계 이동, 상태는 유저 계정에 영속 저장 |
| P4 | JD 매칭 진단 & 이력서 갭 체크리스트 | 구직자로서, 내 프로필이 이 공고와 얼마나 맞는지, 뭘 보완해야 하는지 알고 싶다 | 공고 `skill_tags`/`preferred_points`와 내 프로필 태그 비교해 매칭률(%) 표시, 갭 스킬 리스트를 체크리스트 형태로 제공 |
| P5 | 마감임박 지도 위젯 | 구직자로서, 마감 임박 공고와 내가 지원한 곳을 지도에서 한눈에 보고 싶다 | 지도 API(카카오맵/구글맵)에 마감임박(색상 구분)/지원한 곳/관심 공고 마커 표시, `address.geo_location` 기반 좌표, 통근 반경/시간 필터 동작 |
| P6 | 신입 채용 트렌드 인사이트 | 구직자로서, 신입 공고에서 어떤 스킬을 많이 요구하는지 알고 싶다 | `/tags/skills` 대비 신입 공고 `skill_tags` 빈도 집계, TOP10 통계 시각화 |

### 5-1. 자격요건 뱃지 추출 (P0-신입 맞춤 필터 부속 기능 — 상세 스펙)

**목적**: 공고의 자격요건(`detail.requirements`)을 처음부터 끝까지 읽지 않아도 핵심 조건(전공/재학상태/포트폴리오 제출 여부/툴 등)을 한눈에 파악할 수 있도록 뱃지 형태로 요약 노출한다.

**스코프**:
- 파싱 대상은 **`detail.requirements`(자격요건) 한 필드로 한정**한다.
- `detail.preferred_points`(우대사항)는 뱃지화하지 않고 원문 텍스트로만 노출한다. 우대사항까지 뱃지로 만들면 "우대"를 "필수"로 오인시킬 위험이 있어 명시적으로 스코프 제외한다.
- 모든 뱃지는 출처가 `requirements` 한 곳으로 고정되므로 별도 필수/우대 구분 태그 없이 단일 스타일(예: 실선 뱃지)로 렌더링한다.

**데이터 소스 우선순위** (텍스트 파싱보다 이미 구조화된 필드를 항상 우선 사용):
1. **경력 연차** → 이미 구조화된 `annual_from`/`annual_to` 사용 (P1의 기존 로직 재사용, 텍스트 파싱 불필요)
2. **기술/툴 스택** → 공고에 이미 붙어있는 `skill_tags`(`{id, title}`)를 우선 확인. 여기 없는 툴만 텍스트 파싱으로 보완
3. **그 외** (전공, 재학/졸업 상태, 포트폴리오 제출 필수 여부, 자격증/어학 등 API에 구조화 필드가 없는 항목) → `requirements` 텍스트를 줄 단위로 파싱해서 추출

**툴/기술 사전 구축 방법** (직접 타이핑한 목록이 아니라 API 데이터 기반으로 만든다):
- **직군 스코프 수집**: `/tags/categories`에서 디자인 관련 하위 태그 ID를 얻어 `category_tags` 파라미터로 `/jobs`를 페이지네이션 조회, 해당 직군 공고의 `skill_tags`를 빈도 집계해 "직군별 사전"을 만든다 (전 직군을 한 번에 긁지 않음 — AWS/Java 등 무관한 스킬이 섞이는 것 방지)
- **보완 수집**: `/search/position?query=` 로 직무명(예: "UX/UI 디자이너")을 검색해 category_tags 필터로 누락된 공고를 보완한다
- **requirements 후보 추출**: "사용/활용/숙련/능숙/경험/가능" 등 조건 문맥 단어가 함께 있는 경우에만 툴 이름 후보를 추출한다 (툴 이름이 등장한다고 무조건 추출하지 않음)
- **검증**: 추출한 후보를 `/tags/skills?keyword=` 로 조회해 실제 존재하는 태그인지 확인하고 `verified: true/false`로 구분해 사전에 기록한다. 미검증(`verified: false`) 항목도 사내 툴/신기술일 수 있으므로 폐기하지 않고 낮은 우선순위로만 사용한다
- **저장 형태(v1)**: 위 배치 수집 결과를 **주기적으로 수동 재생성하는 정적 JSON 사전 파일**로 관리한다. 실시간 서빙 시점에는 이 파일만 참조하고 `/tags/skills` 등 외부 API를 호출하지 않는다 (지연시간/쿼터 문제 방지). DB 테이블화·자동 배치 스케줄러는 v1 범위 밖 — 실사용 데이터가 쌓여 필요성이 확인되면 v2에서 검토한다

**파싱 절차**:
1. `requirements` 문자열을 개행(`\n`)으로 split (`•`, `·`, `▪`, `○` 등 불릿 문자도 개행으로 치환)
2. 각 줄 선행부의 `-`, `•` 등 불릿 문자만 제거 후 트림 (`UX/UI`, `0-2년`처럼 단어 내부의 `-`는 보존)
3. 아래 카테고리별 매칭 함수를 각각 독립적으로 적용한다 (하나의 거대 정규식으로 합치지 않음 — 규칙 수정/테스트 용이성 목적)
4. **불리언 게이팅**으로 뱃지 채택 여부를 결정한다: `skill_tags`에 이미 있으면 무조건 채택 / 없으면 "카테고리 정규식 매칭 + (툴의 경우) 조건 문맥 단어 존재 + 같은 줄에 '우대'·'선호' 표현 없음"을 모두 만족할 때만 채택. 가중치 점수 시스템은 v1에서 도입하지 않는다 — 실사용 데이터 없이 점수를 튜닝하는 것은 근거 없는 정밀도를 흉내내는 것에 가깝고, 단순 불리언 규칙으로도 동일한 목적(오탐 방지)을 달성할 수 있다
5. **매칭 안 되는 줄은 뱃지로 만들지 않고 원문 그대로 "기타 자격요건" 영역에 노출한다** (뱃지 신뢰도가 텍스트 커버리지보다 우선 — 억지 뱃지화 금지)

**카테고리 규칙 (v1)**:

| 카테고리 | 매칭 규칙 | 예시 원문 → 뱃지 |
|---|---|---|
| 전공 | `전공자\|관련\s*전공\|전공\s*재학생\|전공\s*졸업자` 등 "전공" 앞뒤 고정 표현만 매칭 (자유 형태 `(.+?)\s*전공`은 오탐 위험이 커서 사용하지 않음) | "디자인, UX/UI 관련 전공 재학생..." → `디자인·UX/UI 전공` |
| 재학/졸업 상태 | `재학\s*생\|재학\s*중` / `졸업\s*예정` | "...재학생 또는 졸업 예정자" → `재학생/졸업예정` |
| 포트폴리오 제출 필수 | `포트폴리오` 존재 + 같은 줄에 `필수\|반드시\|제출해\s*주세요\|미제출\s*시` 존재 (두 조건 다 필요) | "포트폴리오 제출 필수(개인 프로젝트...)" → `포트폴리오 제출 필수` |
| 툴/기술 (skill_tags 미포함분만) | 위에서 구축한 정규화 사전(별칭 포함, 예: figma/피그마) 매칭 + 조건 문맥 단어(사용/활용/숙련/능숙/경험/가능) 동반 | "Figma 등 디자인 툴 사용 가능" → `Figma 가능` |
| 자격증/어학 (v2 예정) | 자체 유지보수 자격증명 사전 매칭 (정보처리기사, 토익 등) | - (v1 범위 아님) |

규칙에 걸리지 않는 문장(예: "디지털 프로덕트 디자인에 대한 관심과 성장 의지 보유", "논리적 사고 및 원활한 커뮤니케이션 능력" 같은 추상적/소프트스킬 문장)은 의도적으로 뱃지화하지 않는다.

**알려진 한계 (v1에서 해결하지 않고 명시만 함)**:
- "당사는 Figma를 사용합니다"(회사 소개 문장)와 "Figma 사용 가능자"(지원자 조건)는 둘 다 동일한 문맥 단어("사용")를 포함해 규칙만으로 완전히 구분되지 않는다. 주어(회사 vs 지원자) 판별은 v1 규칙 기반 파싱의 범위를 넘어서는 문제로, 오탐 사례가 보이면 카테고리 규칙을 개별 보강한다
- 전공 추출 시 결과 문자열이 과도하게 길어지는 경우의 명확한 컷오프 기준은 아직 정하지 않았다 — 실제 공고 샘플을 보면서 임계값을 정한다
- 툴 별칭 사전은 신규 툴/표기가 계속 나오므로 지속적인 수동 갱신이 필요하다 (일회성 구축으로 끝나지 않음)

**품질 검증 (v1)**: 정식 Precision/Recall 측정용 대규모 라벨링 세트(직군별 수백 건)는 v1에서 구축하지 않는다. 대신 배포 전 직군별 20~30건 정도를 수동으로 훑어보는 스팟체크로 명백한 오탐(우대사항이 필수로 표시되는 등)만 걸러내고, 실사용 이후 발견되는 오탐 사례를 카테고리 규칙에 반영하는 방식으로 반복 개선한다. 정식 평가 지표 도입은 v2 이후 실사용 데이터가 쌓였을 때 검토한다.

**Acceptance Criteria**:
- `requirements` 텍스트 중 위 카테고리 규칙에 매칭되는 항목은 뱃지 UI로 렌더링된다
- 매칭되지 않는 문장은 뱃지가 아닌 원문 텍스트로 그대로 노출된다 (정보 누락 없음)
- `preferred_points`는 뱃지 파이프라인에 전혀 포함되지 않는다 (원문 텍스트로만 노출)
- `skill_tags`에 이미 존재하는 툴/기술은 `requirements` 텍스트 파싱으로 중복 추출하지 않는다
- 같은 줄에 "우대"·"선호" 표현이 함께 있으면 해당 줄은 뱃지화하지 않는다
- 툴/기술 사전은 정적 JSON 파일로 관리하며, 실시간 요청 처리 중에는 외부 API(`/tags/skills` 등)를 호출하지 않는다

### 5-2. 회원가입 & 프로필/포트폴리오 (P0) 상세 스펙

**목적**: 원티드 API가 제공하지 않는 사용자 식별/개인화 상태(관심 태그, 지원 현황 등)를 위한 자체 계정 시스템을 만든다. P3/P4/P5의 선행 조건.

**데이터 모델 (자체 DB, Wanted API 미제공)**:
- `users`: id, email, password_hash, created_at
- `profiles`: user_id, career_years(0~2 정수)
- `profile_category_tags`: user_id, category_tag_id — `/tags/categories` 값을 그대로 참조(다중 선택)
- `profile_skill_tags`: user_id, skill_tag_id, skill_tag_title — `/tags/skills?keyword=` 자동완성으로 검색해 선택(다중 선택)
- `portfolios`: user_id, type(url/file), url 또는 file_key

**AC**:
- 이메일 중복 가입 방지, 비밀번호 정책 적용
- 경력은 0~2 정수 범위로만 입력 가능(그 이상 값은 UI에서 원천 차단 — 스코프상 주 타깃 외 사용자 방지)
- 관심 스킬 태그는 자유 텍스트가 아니라 `/tags/skills` 자동완성 결과에서 선택 → skill_tag_id로 저장(제목 문자열 매칭이 아니므로 P4 매칭 로직에서 정확도 보장)
- 관심 직군 태그는 `/tags/categories` 트리에서 선택
- 포트폴리오는 URL 또는 파일 업로드 중 최소 1개 이상 등록
- 마이페이지에서 프로필 수정 가능

**미정 사항 (구현 착수 전 결정 필요)**:
- 인증 방식(세션 vs JWT), 소셜 로그인 도입 여부
- 포트폴리오 파일 저장소(S3 등) 및 허용 형식/최대 용량

### 5-3. 회사 정보(연봉/복지) (P1) 상세 스펙

**API 체이닝**: 공고 상세(`/jobs/{job_id}`) 응답의 `company.registration_number` → `/insight/company?biz_number=`로 재조회. 공고 API 자체에는 연봉 통계가 없어 반드시 2차 호출이 필요하다.

**표시 데이터**:
- 연봉: `averageSalary`(평균연봉), `hiredSalary`(신규입사자 평균연봉). 부가 정보로 `employeeCountNPS`/`employeeCountEI`(가입자 수 기반 인원 추정치), `hireRate`/`leftRate`(입/퇴사율)도 노출 가능
- 복지: `detail.benefits`(자유텍스트, 원문 그대로 노출) + `attraction_tags`(이미 구조화되어 있어 파싱 없이 그대로 뱃지 렌더링)

**AC**:
- `/insight/company` 503/지연 발생 시 화면 전체가 깨지지 않고 "회사 재무 정보 준비중" 폴백 문구 노출
- `company.registration_number`가 없는 공고는 연봉 섹션 자체를 숨김(있을 수 있는 케이스인지 실제 응답으로 확인 필요)
- `attraction_tags`는 별도 파싱 없이 그대로 뱃지 UI로 렌더링

**리스크**: `/insight/company`는 원티드 자체 API가 아닌 별도 백엔드 연동으로 보여 안정성이 낮음 — 실 계정으로 호출량 제한/쿼터 확인 필요

### 5-4. 용어 툴팁 (P2) 상세 스펙

**데이터 소스**: `skill_tags`는 `{id, title}`만 제공하고 설명 문장은 API에 없다. 5-1의 툴 사전과 달리 이건 "설명 콘텐츠" 자체라 API로 대체할 방법이 없어 **100% 자체 제작**해야 한다.

**구축 우선순위**: 전체 스킬을 한 번에 다 채우려 하지 않고, P6(트렌드 인사이트)에서 집계하는 "신입 공고에 자주 등장하는 스킬 TOP N"부터 설명을 채운다 (사용 빈도 기반 우선순위 — 두 기능이 같은 집계 결과를 공유).

**AC**:
- `skill_tags` 각 항목에 hover(PC)/tap(모바일) 시 툴팁 노출
- 사전은 `skill_tag_id` 기준으로 매핑한다(제목 문자열 매칭 금지 — 동명이의 스킬 방지)
- 사전에 설명이 없는 태그는 툴팁을 노출하지 않는다(빈 툴팁 대신 자연스럽게 생략)

**리스크**: 콘텐츠 제작 리소스 문제 — 초기 커버리지 목표치(예: 상위 50개 스킬)를 정하고 시작해야 함

### 5-5. 지원 여정 칸반보드 (P3) 상세 스펙

**데이터 모델**: `applications` 테이블 — user_id, job_id, status(관심/준비중/지원함/결과대기), company_name_snapshot, due_time_snapshot, skill_tags_snapshot(JSON), created_at, updated_at

**핵심 설계 포인트**: 카드에 표시되는 공고 정보는 **칸반에 추가되는 시점의 값을 스냅샷으로 복사 저장**한다. 원본 공고가 이후 마감·삭제되어도 API 재조회 없이 카드 내용이 유지되어야 하기 때문이다. 진행 단계(status)는 원티드 API에 존재하지 않는 개념이라 100% 자체 DB로 관리한다(PLANNING.md 근거: ATS 진행상태 API는 기업 전용).

**AC**:
- 관심 → 준비중 → 지원함 → 결과대기 4단계 컬럼, 드래그 또는 버튼으로 카드 이동
- 카드에는 스냅샷 시점의 회사명/마감일/스킬태그를 표시하고, 원본 공고 최신 상태와 다를 수 있음을 UI에 표기
- 원본 공고 링크(`url` 필드)는 유지하되 공고가 만료돼 링크가 깨져도 카드 자체는 정상 노출

### 5-6. JD 매칭 진단 & 이력서 갭 체크리스트 (P4) 상세 스펙

**매칭 로직**: 정량적 매칭률은 **`skill_tags`(구조화 데이터)만으로 계산**한다.

```
매칭률 = |공고 skill_tags ∩ 내 프로필 skill_tags| / |공고 skill_tags|
```

`preferred_points`(우대사항, 자유텍스트)는 5-1과 동일한 이유(비구조화 텍스트를 정량 매칭에 넣으면 신뢰도가 떨어짐)로 매칭률 계산에 포함하지 않고, 참고용 원문 텍스트로만 별도 노출한다.

**갭 체크리스트**: 공고 skill_tags 중 내 프로필에 없는 항목을 "이 스킬 채우면 매칭률 오름" 리스트로 제공한다. 체크(완료 표시)는 자체 DB에 사용자별로 저장하되, 체크한다고 프로필의 skill_tags가 자동으로 추가되지는 않는다 — 매칭률을 실제로 올리려면 사용자가 프로필(P0)에 해당 스킬을 명시적으로 추가해야 한다.

**AC**:
- 공고 상세 화면에 매칭률(%) 표시
- 갭 리스트는 스킬 태그 단위로 표시, 체크 상태는 영속 저장
- 프로필에 스킬 태그가 하나도 없는 사용자는 매칭률 대신 "프로필을 채워주세요" 유도 문구 노출

### 5-7. 마감임박 지도 위젯 (P5) 상세 스펙

**API 호출 전략**: `/jobs`(목록)에는 `geo_location`이 없어 지도에 마커를 찍으려면 공고별 `/jobs/{job_id}`를 개별 호출해야 한다. 목록에 있는 모든 공고를 전부 상세 조회하면 호출량이 급증하므로, **실제로 화면(뷰포트/필터링 결과)에 노출되는 공고만** 상세 조회하고 결과를 캐싱한다(좌표/주소는 자주 바뀌지 않으므로 TTL 예: 1일).

**D-day 색상 구분 기준(안, 팀 협의로 조정 가능)**: D-3 이내 빨강, D-7 이내 주황, 그 외 기본색

**통근 필터 v1 범위**: 사용자 현재 위치 기준 **직선 반경(km)** 필터만 v1에 포함한다. 실제 "통근 소요시간" 필터는 별도 길찾기 API(카카오모빌리티 등) 연동이 필요해 v2로 보류한다.

**AC**:
- 마커 클릭 시 공고 요약 팝업(회사명/마감일/스킬태그) 노출
- 마감임박/지원한 곳(P3 연동)/관심 등록 공고를 마커 색상·아이콘으로 구분
- 반경 필터 슬라이더로 마커 좁혀보기 동작

**리스크**: 상세 API 호출량이 지도 위젯의 성능/쿼터를 좌우함 — 캐싱 전략이 선행 조건

### 5-8. 신입 채용 트렌드 인사이트 (P6) 상세 스펙

**집계 로직**: P1에서 정의한 "진짜 신입 가능" 판정 기준과 동일한 기준으로 필터링한 공고를 대상으로 `skill_tags` 출현 빈도를 집계해 TOP10을 낸다. 이 집계 파이프라인은 5-1(툴 사전 구축)에서 이미 만든 "category_tags로 스코프 좁혀 공고 수집 → skill_tags 빈도 집계" 로직을 그대로 재사용한다(신규 구축 불필요).

**주기**: 요청마다 전체 공고를 다시 훑지 않고, 배치(예: 1일 1회) 집계 후 캐싱된 결과를 서빙한다.

**AC**:
- TOP10 스킬을 바 차트/리스트로 시각화
- (선택) 직군 필터와 결합해 "디자인 직군 한정 트렌드"처럼 좁혀보기 — v1 포함 여부는 팀 논의 필요

## 6. 스코프 제외 (Non-goals)

API 검토를 거쳐 아래 2개는 **명시적으로 제외**한다. (근거: PLANNING.md 7절)

- **경쟁 강도(지원자 수)**: 공고 단위 지원자 수를 노출하는 필드가 API에 없음. 확정 제외.
- **합격률 예측**: `/ai/pass/text-prediction/async`가 존재하나 별도 유료 계약 필요(team-ml@wantedlab.com). 데이터 부재가 아닌 계약 이슈. MVP 제외.

## 7. Phase 2 (이번 스코프 밖)

- **기업용 입력 기능**(신입친화 인증 배지, 탈락 피드백, 채용 프로세스 투명성 등): 원티드 기업용 API는 별도 인증 키(`x-wanted-dashboard-service-key`) 체계라 접근 불가. 자체 기업 계정/입력 시스템을 새로 만들어야 하는 양면 플랫폼 확장이라 별도 검토 후 착수.

## 8. 의존 관계

- **P0(회원가입/프로필)** → P3(칸반보드), P4(JD 매칭), P5(지도위젯의 "지원한 곳" 마커)의 선행 조건. 특히 `profile_skill_tags`가 skill_tag_id 기반으로 저장되어야 P4의 매칭률 계산이 정확함(5-2, 5-6 참고)
- **P0(신입 맞춤 필터)** → 모든 화면의 공고 데이터 출처, 사실상 전체 기능의 기반
- **5-1(자격요건 뱃지) 사전 구축 파이프라인** → P6(트렌드 인사이트)의 집계 로직이 동일 파이프라인을 재사용(5-8 참고). 5-1을 먼저 만들면 P6 구현 비용이 줄어듦
- **P6(트렌드 인사이트) 집계 결과** → P2(용어 툴팁)의 콘텐츠 제작 우선순위(어떤 스킬부터 설명을 채울지) 입력값으로 사용(5-4 참고)
- **P3(칸반보드)** → P5(지도위젯)의 "내가 지원한 곳" 마커 데이터 출처

## 9. 리스크 / 확인 필요 사항

- `/insight/company` 안정성 및 쿼터 — 실제 계정으로 테스트 필요 (503 케이스 문서화됨)
- `skill_tags`에 설명 필드가 없어 용어 사전은 100% 자체 구축 — 콘텐츠 제작 리소스 필요
- 지도 위젯은 목록 API에 `geo_location`이 없어 공고별 상세 API 추가 호출 필요 — API 호출량/성능 검토 필요
- 자격요건 뱃지(5-1절)의 카테고리 규칙은 기업마다 자유 형식으로 작성되어 커버리지가 100%가 안 될 수 있음 — 규칙셋은 실제 공고 샘플로 지속 튜닝 필요
- 기업이 우대성 문장을 `requirements` 필드에 잘못 적어넣는 등 원본 데이터 자체의 인적 오류는 파싱 로직으로 해결 불가 (알려진 한계로 문서화만)
- P0 인증 방식(세션/JWT, 소셜로그인 여부)과 포트폴리오 파일 저장소가 아직 미정 — 착수 전 기술 스택 결정 필요(5-2 참고)
- P4 매칭률은 `skill_tags`로만 계산하기로 했으므로, 프로필에 스킬 태그를 안 채운 사용자는 매칭 기능 자체가 무의미함 — 온보딩 단계에서 스킬 태그 입력을 유도하는 UX가 필요
- P5 지도 위젯은 공고별 상세 API 개별 호출 + 캐싱이 선행되어야 하며, 캐시 TTL/무효화 정책이 아직 확정되지 않음
- P6 트렌드 집계와 P2 용어사전 우선순위가 같은 배치 파이프라인 결과에 의존하므로, 이 파이프라인이 지연되면 두 기능 모두 영향받음(8절 의존관계 참고)
