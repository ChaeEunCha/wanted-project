# PRD — 원티드 API 기반 신입 구직자 대시보드

> 이 문서는 [PLANNING.md](./PLANNING.md)에서 정리한 기획 내용을 제품 요구사항 형태로 재구성한 문서입니다. 기능별 배경/근거는 PLANNING.md를 참고하고, 여기서는 요구사항·우선순위·완료 기준을 다룹니다.

## 1. 개요

- **서비스명**: 원티드 API 기반 대시보드
- **한 줄 정의**: 원티드 오픈API 데이터를 재구성해서, 처음 구직하는 청년이 겪는 정보 비대칭과 파편화 문제를 줄여주는 개인화 대시보드
- **데이터 소스**: Wanted OpenAPI v1/v2 (client_id/secret 인증). 일부 기능은 자체 DB(회원/프로필/지원상태)로 보완

## 2. 배경 및 문제 정의

처음 구직자는 아래 문제를 겪는다:

1. 신입 태그 공고인데 실제 요구 경력이 1~3년인 경우가 많아 헷갈림
2. 공고 속 기술 스택/직무 용어 장벽
3. 공고 홍수 속에서 마감일을 놓침
4. 지원 이력이 여기저기 파편화됨
5. 회사 정보(연봉/복지)를 알기 어려움
6. 내 역량이 공고 기준에 맞는지 판단이 어려움
7. (경쟁강도, 합격률을 모르는 문제도 있었으나 API 검토 결과 충족 불가 — 6절 참고)

## 3. 목표

- 신입 구직자가 **가짜 신입 공고를 걸러내고**, **용어를 몰라도 지원 여부를 판단**할 수 있게 한다
- 지원 과정(관심→지원→결과)을 **한 곳에서 관리**할 수 있게 한다
- 지원 전에 **회사 정보(연봉/복지)**를 확인할 수 있게 한다
- 내 역량과 공고 요구사항의 **갭을 액션으로 연결**한다

## 4. 타깃 유저 / 페르소나

- **주 타깃**: 경력 0년(완전 신입) 구직자
- **부 타깃**: 1~2년차 주니어 (경력 필터/매칭 로직 범위에 포함하되 UI 우선순위는 신입에 맞춤)

## 5. 스코프 — 제작 기능 (8개, 우선순위 순)

우선순위 기준: **0번·1번은 병렬 최우선 착수** (하나는 계정/DB 인프라, 하나는 API 연동 코어이며 이후 모든 기능이 이 둘에 의존).

| 우선순위 | 기능 | User Story | 완료 기준 (Acceptance Criteria) |
|---|---|---|---|
| P0 | 회원가입 & 프로필/포트폴리오 | 신입 구직자로서, 내 경력(0~2년)과 관심 태그·포트폴리오를 등록해서 이후 기능들이 나를 알아보게 하고 싶다 | 회원가입/로그인 가능, 경력 0~2년 범위 선택 UI, 관심 직군/스킬 태그 등록, 포트폴리오 링크/파일 등록. 이 데이터가 DB에 저장되어 4·6·3번 기능에서 재사용됨 |
| P0 | 신입 맞춤 필터 대시보드 | 신입 구직자로서, 진짜 신입이 지원 가능한 공고만 보고 싶다 | `annual_from`/`annual_to` 파싱해 "진짜 신입 가능" 여부 뱃지 표시, 직군/지역 조합 필터 동작 |
| P1 | 회사 정보(연봉/복지) | 구직자로서, 지원 전에 이 회사의 평균연봉과 복지를 확인하고 싶다 | 공고 상세에서 `company.registration_number` → `/insight/company` 체이닝으로 `averageSalary`/`hiredSalary` 표시, `detail.benefits` 텍스트 + `attraction_tags` 표시. `/insight/company` 503 응답 시 "정보 없음" 폴백 처리 |
| P2 | 용어 툴팁 | 구직자로서, 공고 속 낯선 기술/직무 용어에 마우스를 올리면 쉬운 설명을 보고 싶다 | `skill_tags` 각각에 툴팁 UI, 자체 구축한 용어 사전 데이터와 매칭되어 설명 노출 (매칭 안 되는 태그는 툴팁 미노출) |
| P3 | 지원 여정 칸반보드 | 구직자로서, 내가 관심 있거나 지원한 공고를 단계별로 관리하고 싶다 | 관심/준비중/지원함/결과대기 4단계 칸반, 카드에 회사명·마감일·스킬태그 스냅샷 표시, 드래그 또는 버튼으로 단계 이동, 상태는 유저 계정에 영속 저장 |
| P4 | JD 매칭 진단 & 이력서 갭 체크리스트 | 구직자로서, 내 프로필이 이 공고와 얼마나 맞는지, 뭘 보완해야 하는지 알고 싶다 | 공고 `skill_tags`/`preferred_points`와 내 프로필 태그 비교해 매칭률(%) 표시, 갭 스킬 리스트를 체크리스트 형태로 제공 |
| P5 | 마감임박 지도 위젯 | 구직자로서, 마감 임박 공고와 내가 지원한 곳을 지도에서 한눈에 보고 싶다 | 지도 API(카카오맵/구글맵)에 마감임박(색상 구분)/지원한 곳/관심 공고 마커 표시, `address.geo_location` 기반 좌표, 통근 반경/시간 필터 동작 |
| P6 | 신입 채용 트렌드 인사이트 | 구직자로서, 신입 공고에서 어떤 스킬을 많이 요구하는지 알고 싶다 | `/tags/skills` 대비 신입 공고 `skill_tags` 빈도 집계, TOP10 통계 시각화 |

## 6. 스코프 제외 (Non-goals)

API 검토를 거쳐 아래 2개는 **명시적으로 제외**한다. (근거: PLANNING.md 7절)

- **경쟁 강도(지원자 수)**: 공고 단위 지원자 수를 노출하는 필드가 API에 없음. 확정 제외.
- **합격률 예측**: `/ai/pass/text-prediction/async`가 존재하나 별도 유료 계약 필요(team-ml@wantedlab.com). 데이터 부재가 아닌 계약 이슈. MVP 제외.

## 7. Phase 2 (이번 스코프 밖)

- **기업용 입력 기능**(신입친화 인증 배지, 탈락 피드백, 채용 프로세스 투명성 등): 원티드 기업용 API는 별도 인증 키(`x-wanted-dashboard-service-key`) 체계라 접근 불가. 자체 기업 계정/입력 시스템을 새로 만들어야 하는 양면 플랫폼 확장이라 별도 검토 후 착수.

## 8. 의존 관계

- **P0(회원가입/프로필)** → P3(칸반보드), P4(JD 매칭), P5(지도위젯의 "지원한 곳" 마커)의 선행 조건
- **P0(신입 맞춤 필터)** → 모든 화면의 공고 데이터 출처, 사실상 전체 기능의 기반

## 9. 리스크 / 확인 필요 사항

- `/insight/company` 안정성 및 쿼터 — 실제 계정으로 테스트 필요 (503 케이스 문서화됨)
- `skill_tags`에 설명 필드가 없어 용어 사전은 100% 자체 구축 — 콘텐츠 제작 리소스 필요
- 지도 위젯은 목록 API에 `geo_location`이 없어 공고별 상세 API 추가 호출 필요 — API 호출량/성능 검토 필요
